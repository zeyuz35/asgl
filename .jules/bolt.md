## 2024-03-01 - Optimizing PCA for Sparse Matrices
**Learning:** scikit-learn's `PCA(n_components="some_fraction")` does not natively support sparse matrices because `svd_solver='full'` requires dense conversions. When `AdaptiveWeights` tries to achieve `variability_pct < 1` for sparse data by iteratively incrementing `n_comp` from `1` to `max_comp` and re-running PCA, it implicitly triggers a dense conversion on every loop iteration, leading to massive memory and performance overhead (23s+ for small inputs).
**Action:** Replace iterative PCA incrementing with a single run of `PCA(n_components=max_comp, svd_solver="arpack")` to calculate all principal components at once without iterative dense conversions. Then compute the cumulative explained variance manually to truncate components.

## 2026-03-07 - CVXPY DPP Compliance with Parameterized Norms
**Learning:** In CVXPY, expressions like `cp.norm1(cp.multiply(param, beta))` or multiplying a parameter directly with a norm (e.g., `lambda_param * cp.norm1(beta)`) are flagged as non-DPP (Disciplined Parametrized Programming). This causes CVXPY to emit a UserWarning and triggers a full, unoptimized rebuild/canonicalization of the problem upon every solve. Since the `Regressor` instantiates a new `cp.Problem` for each `fit` call instead of reusing one, explicitly wrapping `lambda1` or weights in `cp.Parameter` offers zero caching benefit while adding massive overhead and compilation warnings compared to just multiplying by native floats or numpy arrays.
**Action:** When defining parameterized objectives in CVXPY where the problem structure is rebuilt from scratch per call anyway, use raw Python scalars or NumPy arrays directly for hyperparameters and weights instead of `cp.Parameter` to bypass unnecessary DPP checks, avoid warnings, and significantly speed up compilation and solve times.

## 2026-03-11 - [Vectorized Group Weight Aggregation]
**Learning:** `np.linalg.norm` iteratively computed in a Python `for` loop across masks created from a large dictionary mapping scales very poorly as O(N*G) where N is number of features and G is number of groups. We discovered that calculating grouped 2-norms for adaptive weights (`fit_weights`) can bottleneck model fitting severely when dealing with many features/groups.
**Action:** Replace `for` loops that compute group statistics by masking, with vectorized indices using `np.argsort` followed by `np.unique(..., return_index=True)` and grouping the aggregated statistics with `np.add.reduceat`. This brings complexity to O(N log N) effectively reducing calculation times from seconds/minutes to milliseconds on typical data scales.

## 2026-03-22 - Optimizing PLSRegression iterative re-fitting
**Learning:** For adaptive penalization techniques like Adaptive Lasso, when calculating adaptive weights using `pls_pct` (Partial Least Squares percentage variance), the original code fit a `PLSRegression` model to identify the number of components $k$ needed to reach the desired variance, and then fit a *second* `PLSRegression(n_components=k)` to obtain the coefficients. However, `PLSRegression` extracts components sequentially. This means that the rotations and loadings for the first $k$ components are identical regardless of whether the model was fit for $k$ components or $P$ components. Fitting a second PLS model is completely redundant and incurs O(N * P * k) time complexity unnecessarily.
**Action:** Instead of re-instantiating and re-fitting a second `PLSRegression(n_components=k)` model, directly calculate the required `coef_` by multiplying the `x_rotations_` and `y_loadings_` from the original full fit, sliced up to $k$ components: `coef_ = np.dot(pls.x_rotations_[:, :k], pls.y_loadings_[:, :k].T)`. This transforms a second model fitting process into a lightning fast, single O(N_features * k * N_outputs) matrix multiplication.
