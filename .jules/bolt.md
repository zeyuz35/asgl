## 2024-03-01 - Optimizing PCA for Sparse Matrices
**Learning:** scikit-learn's `PCA(n_components="some_fraction")` does not natively support sparse matrices because `svd_solver='full'` requires dense conversions. When `AdaptiveWeights` tries to achieve `variability_pct < 1` for sparse data by iteratively incrementing `n_comp` from `1` to `max_comp` and re-running PCA, it implicitly triggers a dense conversion on every loop iteration, leading to massive memory and performance overhead (23s+ for small inputs).
**Action:** Replace iterative PCA incrementing with a single run of `PCA(n_components=max_comp, svd_solver="arpack")` to calculate all principal components at once without iterative dense conversions. Then compute the cumulative explained variance manually to truncate components.

## 2026-03-07 - CVXPY DPP Compliance with Parameterized Norms
**Learning:** In CVXPY, expressions like `cp.norm1(cp.multiply(param, beta))` or multiplying a parameter directly with a norm (e.g., `lambda_param * cp.norm1(beta)`) are flagged as non-DPP (Disciplined Parametrized Programming). This causes CVXPY to emit a UserWarning and triggers a full, unoptimized rebuild/canonicalization of the problem upon every solve. Since the `Regressor` instantiates a new `cp.Problem` for each `fit` call instead of reusing one, explicitly wrapping `lambda1` or weights in `cp.Parameter` offers zero caching benefit while adding massive overhead and compilation warnings compared to just multiplying by native floats or numpy arrays.
**Action:** When defining parameterized objectives in CVXPY where the problem structure is rebuilt from scratch per call anyway, use raw Python scalars or NumPy arrays directly for hyperparameters and weights instead of `cp.Parameter` to bypass unnecessary DPP checks, avoid warnings, and significantly speed up compilation and solve times.

## 2026-03-09 - Vectorized group norms calculation
**Learning:** In penalization models grouping features (like GL, SGL, ASGL), calculating the L2 norm of weights for each group using a Python `for` loop over groups is slow because the number of groups $G$ and features $N$ can be large ($O(N \cdot G)$ bottleneck).
**Action:** Replace the Python loop with a vectorized approach: sort the indices using `np.argsort`, use `np.unique` to find group start indices, square the values, and sum them per group in a single vectorized pass using `np.add.reduceat`. Finally, take the square root. This brings the time complexity down to $O(N \log N)$ and provides a ~5x speedup for large datasets.
